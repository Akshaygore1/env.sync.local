Secure-peer trust model

  • No global root CA shipped; trust is deployment-local with pinned certificates.
  • Peer registry tracks pending/approved/revoked states.
  • On approve/revoke, signed membership events replicate across peers.
  • Replay protection uses monotonic event IDs plus timestamp checks.

Threat-model highlights

trusted-owner-ssh

  • Strength: mature SSH transport security and operational simplicity.
  • Limitation: compromise of one fully trusted peer can impact others due to broad trust.

secure-peer

  • Strength: explicit authorization and mandatory encryption at rest.
  • Strength: mTLS avoids shell access sharing, reducing blast radius vs SSH trust mesh.
  • Operational requirement: invitation/approval workflow and identity material management.

Operational security checklist

  • Keep file permissions strict (600 for secrets).
  • Avoid logging secret values (env-sync logging keeps this boundary).
  • Regularly audit approved peers in secure-peer mode.
  • Use backups and key lifecycle controls during membership changes.
