Differences in modes

Security models you can choose explicitly

Mode A: dev-plaintext-http

Debug-only mode. No encryption at rest, no transport protection, and no authentication.

Mode B: trusted-owner-ssh

Default mode for same-owner devices. SSH provides encrypted transport and authentication. Storage is plaintext by default because trust is already broad in this model; optional encryption can be enabled for defense-in-depth.

Sync strategy

  1. Discover peers with mDNS and SSH reachability checks.
  2. Fetch secrets via SCP/SSH.
  3. Compare metadata versions/timestamps.
  4. Merge and write local file (with backup).

Mode C: secure-peer

Designed for cross-owner collaboration: no shell access needed between peers, mandatory AGE encryption, and mTLS-authenticated HTTPS transfer.

Sync strategy

  1. Discover peers over mDNS.
  2. Establish mTLS and verify authorization.
  3. Fetch encrypted secrets and decrypt locally.
  4. Merge using per-key timestamps.
  5. Re-encrypt to all known recipients and save backup.
  6. Replay signed membership events for offline catch-up.

Choosing the right mode

  • Use trusted-owner-ssh for your own machines where SSH trust already exists.
  • Use secure-peer when different owners collaborate but should not share shell access.
  • Use dev-plaintext-http only in isolated local debugging contexts.
