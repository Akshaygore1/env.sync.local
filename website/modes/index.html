Three modes, one CLI

env-sync gives you explicit control over how secrets are stored, transported, and who gets access. Pick the security model that fits your scenario.

Mode comparison

Feature dev-plaintext-http trusted-owner-ssh secure-peer
Storage encryption Plaintext Plaintext (optional AGE) AGE encrypted (mandatory)
Transport security HTTP (plaintext) SCP / SSH HTTPS + mTLS
Authentication None SSH keys Mutual TLS certificates
Authorization Open to all peers Implicit via SSH trust Explicit approval required
Onboarding Automatic discovery Zero-touch if SSH configured Invitation-based
Use case Local debugging only Personal device fleet Cross-team collaboration

Mode A : dev-plaintext-http

Purpose: Quick local debugging with test data

This mode provides zero security — no encryption, no authentication. Secrets are transmitted and stored in plaintext. Use exclusively for throwaway testing environments with fake credentials.

Technical details

  • Storage: Plaintext .secrets.env file
  • Transport: HTTP on port 5739
  • Discovery: mDNS broadcasts on local network
  • Access control: None — any peer can sync
Warning: Never use this mode with real credentials. All data is visible to anyone on your network.
enable dev mode 
env-sync mode set dev-plaintext-http

Mode B : trusted-owner-ssh default

Purpose: Sync secrets across your own machines

Ideal when all devices belong to you. Leverages existing SSH infrastructure for encrypted transport and authentication. Storage is plaintext by default since trust is already broad in this model — optional AGE encryption available for additional protection.

Technical details

  • Storage: Plaintext by default, AGE encryption optional
  • Transport: SCP over SSH (encrypted)
  • Discovery: mDNS, filtered by SSH reachability test
  • Authentication: SSH key pairs (your existing ~/.ssh keys)
  • Authorization: Implicit — if you have SSH access, you can sync

Sync workflow

  1. Discover peers via mDNS announcements
  2. Test SSH connectivity to each discovered peer
  3. Fetch .secrets.env from reachable peers via SCP
  4. Compare file versions using metadata timestamps
  5. Merge changes with per-key conflict resolution
  6. Write updated file locally with automatic backup
enable trusted-owner mode 
env-sync mode set trusted-owner-ssh

Mode C : secure-peer

Purpose: Share secrets across teams without shell access

Designed for collaboration between different owners. No SSH access required or shared. Uses mTLS for transport authentication and AGE for at-rest encryption. Every peer must be explicitly invited and approved.

Technical details

  • Storage: AGE encrypted (mandatory)
  • Transport: HTTPS with mutual TLS (client + server certificates)
  • Discovery: mDNS announcements
  • Authentication: TLS certificate verification + certificate pinning
  • Authorization: Explicit — peers have pending, approved, or revoked status
  • Encryption: Each secret encrypted to multiple AGE recipients

Sync workflow

  1. Discover peers via mDNS
  2. Establish mTLS connection and verify peer authorization status
  3. Fetch encrypted secrets via HTTPS endpoint
  4. Decrypt using local AGE private key
  5. Merge changes with per-key timestamps
  6. Re-encrypt to all approved recipients (AGE public keys)
  7. Save with automatic backup
  8. Fetch and replay signed membership events for offline peers

Peer management

  • Invitation: Generate time-limited enrollment token
  • Request access: New peer sends token + identity
  • Approval: Existing peer approves, adds to registry
  • Trust propagation: Signed membership events replicate approval across network
  • Revocation: Remove peer access at any time, propagates automatically
enable secure-peer mode 
env-sync mode set secure-peer

Choosing the right mode

Use trusted-owner-ssh when:

  • All machines belong to you
  • You already manage SSH keys across your devices
  • Your threat model accepts plaintext at-rest storage
  • You want zero-configuration setup

Setup guide →

Use secure-peer when:

  • Multiple people need access to the same secrets
  • You don't want to share shell access between machines
  • At-rest encryption is required
  • You need explicit peer approval/revocation

Setup guide →

Use dev-plaintext-http when:

  • Testing env-sync locally
  • Working with fake/throwaway credentials
  • Speed matters more than security
